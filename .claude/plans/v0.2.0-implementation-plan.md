## Phase 1: Core Type & Interface Changes

### 1.1 Extend `PaymentGatewayAdapter` ‚Äî add optional `getPaymentStatus`

```ts
export interface PaymentGatewayAdapter {
  // ... existing ...
  /** Poll gateway for payment status (fallback if webhook delayed). Optional per gateway. */
  getPaymentStatus?(orderId: string): Promise<WebhookStatus>;
}
```

### 1.2 Extend `StorageAdapter` ‚Äî new methods are **optional** (non-breaking)

```ts
export interface StorageAdapter {
  // ... existing required methods ...
  getTransactions?(userId: string, opts: { limit: number; offset: number; type?: 'credit' | 'debit' }): Promise<LedgerEntry[]>;
  getCheckouts?(userId: string, opts: { limit: number; offset: number; status?: CheckoutSession['status'] }): Promise<CheckoutSession[]>;
}
```

In `createLedger` and `createCheckoutManager`, check existence and throw a descriptive error if called but not implemented.

### 1.3 Richer `handleWebhook` return type with reasons

```ts
export type WebhookAction = 'credited' | 'skipped' | 'duplicate';
export type WebhookSkipReason = 'unparseable' | 'non_success_status' | 'session_not_found' | 'already_processed' | 'status_updated';
export interface WebhookResult {
  action: WebhookAction;
  reason?: WebhookSkipReason;
}
```

Updated logic:

- Signature invalid ‚Üí still throws `WebhookVerificationError`
- Parsed null ‚Üí `{ action: 'skipped', reason: 'unparseable' }`
- Non-success status (but failed/expired) ‚Üí update checkout to `'failed'`, return `{ action: 'skipped', reason: 'status_updated' }`
- Non-success status (pending/other) ‚Üí `{ action: 'skipped', reason: 'non_success_status' }`
- Session not found ‚Üí `{ action: 'skipped', reason: 'session_not_found' }`
- Session already processed ‚Üí `{ action: 'skipped', reason: 'already_processed' }`
- Idempotency conflict ‚Üí `{ action: 'duplicate' }`
- Successfully credited ‚Üí `{ action: 'credited' }`

### 1.4 Handle expired/failed webhook statuses

When `parsed.status` is `'expired'` or `'failed'`, update the checkout status to `'failed'` if currently `'pending'`. This prevents stale `pending` checkouts from lingering forever.

### 1.5 Configurable fetch timeout ‚Äî on ALL gateway config objects

```ts
// Applied to MidtransConfig, XenditConfig
timeoutMs?: number; // Default: 30_000
```

Use `AbortSignal.timeout(config.timeoutMs ?? 30_000)` on **all** fetch calls (createCheckout, getPaymentStatus), not just createCheckout.

### 1.6 Delete `schemas.ts`

`packages/core/src/schemas.ts` exports 3 plain type aliases (`SpendParams`, `TopUpParams`, `CheckoutParams`) that are unused, have no Zod, and mislead contributors. Delete the file and remove its re-export from `index.ts`.

### 1.7 Update `createMockStorage()` in test helpers

Add `getTransactions` and `getCheckouts` implementations to the mock so existing tests continue to compile after the interface change.

### 1.8 Add `Wallet.getTransactions` and `Wallet.getCheckouts`

Expose both query methods on the `Wallet` interface so users can build "my usage" and "my purchases" screens from a single object.

## Phase 2: Xendit Gateway Adapter (`packages/gateway-xendit`)

### 2.1 Identity scheme (CRITICAL ‚Äî reviewer fix)

`CheckoutSession.id` = `external_id` (user-controlled, format: `${userId}-${randomUUID()}`). This is consistent with Midtrans where `CheckoutSession.id = orderId`. The Xendit-generated `invoice_id` is NOT stored as the session ID.

For `getPaymentStatus`, use `GET /v2/invoices?external_id=<id>` to look up by external_id.

### 2.2 `XenditConfig`

```ts
export interface XenditConfig {
  secretKey: string;        // Xendit API secret key
  callbackToken: string;    // x-callback-token for webhook verification
  sandbox?: boolean;        // Default: true
  timeoutMs?: number;       // Default: 30_000
}
```

### 2.3 `createXenditGateway(config: XenditConfig): PaymentGatewayAdapter`

**`createCheckout`:**

- Generate `externalId = ${params.userId}-${randomUUID()}`
- POST `https://api.xendit.co/v2/invoices` with Basic auth (`secretKey:`)
- Body: `{ external_id: externalId, amount, success_redirect_url, failure_redirect_url, currency: 'IDR' }`
- Return: `{ id: externalId, redirectUrl: response.invoice_url, ... }`
- Apply `AbortSignal.timeout(config.timeoutMs ?? 30_000)`

**`verifyWebhook`:**

- Constant-time comparison: `crypto.timingSafeEqual(Buffer.from(signature), Buffer.from(config.callbackToken))`
- Guard: both buffers must be same length, return false if not

**`parseWebhookPayload`:**

- Extract `{ orderId: payload.external_id, grossAmount: payload.amount }`
- Status mapping: `PAID`/`SETTLED` ‚Üí `success`, `EXPIRED` ‚Üí `expired`, `PENDING` ‚Üí `pending`, else ‚Üí `failed`

**`getPaymentStatus`:**

- GET `https://api.xendit.co/v2/invoices?external_id=<orderId>`
- Map response `status` to `WebhookStatus`
- Apply timeout

### 2.4 Tests (`packages/gateway-xendit/src/__tests__/xendit.test.ts`)

1. `verifyWebhook` ‚Äî valid token, invalid token, different length strings, timing-safe
2. `parseWebhookPayload` ‚Äî all Xendit statuses mapped correctly
3. `parseWebhookPayload` ‚Äî missing fields, non-object, null
4. `parseWebhookPayload` ‚Äî both PAID and SETTLED map to success (Xendit sends both)
5. `createCheckout` ‚Äî mock fetch success, error response
6. `createCheckout` ‚Äî externalId format matches `${userId}-${uuid}` pattern
7. `getPaymentStatus` ‚Äî mock fetch, all status mappings
8. `getPaymentStatus` ‚Äî non-200 throws GatewayError

## Phase 3: Midtrans Improvements

### 3.1 Fix timing-safe webhook verification (SECURITY ‚Äî reviewer fix)

Replace `hash === signature` with:

```ts
import { createHash, timingSafeEqual } from 'node:crypto';
// ...
const hashBuf = Buffer.from(hash, 'hex');
const sigBuf = Buffer.from(signature, 'hex');
if (hashBuf.length !== sigBuf.length) return false;
return timingSafeEqual(hashBuf, sigBuf);
```

### 3.2 Add dual base hosts (reviewer fix)

The Snap API uses `app.sandbox.midtrans.com`, but the Status API uses `api.sandbox.midtrans.com`. Add a second host variable:

```ts
const snapHost = sandbox ? 'app.sandbox.midtrans.com' : 'app.midtrans.com';
const apiHost = sandbox ? 'api.sandbox.midtrans.com' : 'api.midtrans.com';
```

### 3.3 Extract `statusMap` to module scope

Move the status map from inside `parseWebhookPayload` to module scope so both `parseWebhookPayload` and `getPaymentStatus` can share it.

### 3.4 Add `getPaymentStatus`

- GET `https://${apiHost}/v2/${orderId}/status` with Basic auth
- Reuse extracted `statusMap` to map `transaction_status`
- Apply `AbortSignal.timeout(config.timeoutMs ?? 30_000)`
- Non-200 ‚Üí throw `GatewayError`

### 3.5 Add `timeoutMs` to `MidtransConfig`

Apply timeout to existing `createCheckout` fetch AND new `getPaymentStatus` fetch.

### 3.6 Tests

1. `getPaymentStatus` ‚Äî mock fetch, all 6 status mappings
2. `getPaymentStatus` ‚Äî non-200 throws GatewayError
3. `verifyWebhook` ‚Äî verify timing-safe (existing tests still pass with new implementation)

## Breaking Changes (v0.x ‚Äî documented in CHANGELOG)

1. **`handleWebhook` return type**: `Promise<void>` ‚Üí `Promise<WebhookResult>` ‚Äî callers ignoring return are unaffected
2. **`StorageAdapter` interface**: two new **optional** methods ‚Äî existing custom adapters compile without changes; calling the new methods without implementing them throws a descriptive error
3. **`schemas.ts` removed**: `SpendParams`, `TopUpParams`, `CheckoutParams` types deleted ‚Äî these were unused; anyone importing them needs to define locally
