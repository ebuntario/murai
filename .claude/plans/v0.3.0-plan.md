# v0.3.0 Implementation Plan — Developer Experience

## Goal

Any developer can integrate token-wallet into a real Next.js app by following a documented example and reference site — without reading the source code.

## Decisions (resolved)

| Question | Decision | Rationale |
| --- | --- | --- |
| Deployment target | **GitHub Pages** | Free, zero vendor lock-in, sufficient for pre-1.0. Add Vercel later if needed. |
| API reference strategy | **Hand-written** | API surface is small (6 factory functions, 5 error classes). Hand-written docs are higher quality and the maintenance cost is negligible at this scale. |
| Example app DB | **Document Neon free tier as suggested option** | Reader can use any PostgreSQL. Example README mentions Neon as zero-setup option. |
| Domain | **GitHub Pages default** | No custom domain until there's traffic. Can add later without breaking anything. |
| Number of example apps | **One (Next.js + Midtrans)** | One well-maintained example > two half-maintained ones. Hono integration shown as a code snippet in docs. |
| Doc framework | **Starlight (Astro)** | Purpose-built for library docs. Zero JS by default, built-in Pagefind search, dark mode, sidebar, MDX. Well-maintained, large community. |

### Code Example Validation Strategy

**Problem:** Docs code snippets drift from source after breaking changes.

**Solution:** Use Starlight's `<Code>` component with Vite `?raw` imports to source examples from real `.ts` files that are typechecked in CI.

```text
docs/
└── src/
    ├── examples/                    # Real .ts files, typechecked
    │   ├── quickstart-setup.ts      # Wallet wiring example
    │   ├── webhook-handler.ts       # Webhook endpoint example
    │   ├── spend-tokens.ts          # Spend flow example
    │   └── query-history.ts         # Transaction query example
    └── content/docs/
        └── getting-started/
            └── quickstart.mdx       # Imports from ../examples/ via ?raw
```

In MDX pages:

```mdx
import { Code } from '@astrojs/starlight/components';
import setupCode from '../../examples/quickstart-setup.ts?raw';

<Code code={setupCode} lang="ts" title="lib/wallet.ts" />
```

The `docs/src/examples/` directory imports from `token-wallet` (the published package). These files are real TypeScript and get typechecked as part of the docs build. If a breaking change lands in v0.4.0, the docs build breaks in CI — not silently in production.

## Deliverable 2: Example App (`examples/nextjs/`)

### Tech Stack

- Next.js 15 (App Router)
- React Server Components
- Tailwind CSS
- Drizzle ORM + PostgreSQL
- `token-wallet` meta-package (installed from npm, NOT workspace link)

### Structure

```text
examples/nextjs/
├── package.json                # Standalone — NOT a workspace package
├── .env.example                # All required vars documented
├── drizzle.config.ts
├── src/
│   ├── app/
│   │   ├── layout.tsx
│   │   ├── page.tsx            # Dashboard: balance + top-up + spend
│   │   ├── api/
│   │   │   └── webhooks/
│   │   │       └── midtrans/
│   │   │           └── route.ts    # POST webhook handler
│   │   └── success/
│   │       └── page.tsx        # Post-payment redirect
│   ├── lib/
│   │   └── wallet.ts           # Wallet + checkout singleton setup
│   └── components/
│       ├── balance-display.tsx
│       ├── topup-button.tsx
│       └── spend-button.tsx
├── README.md                   # Step-by-step: clone → env → db → run
└── .gitignore
```

### What it demonstrates

- Full `createWallet` + `createMidtransGateway` + `createDrizzleStorage` wiring
- Server Action for initiating top-up → redirect to Midtrans Snap
- Route Handler for webhook handling (`POST /api/webhooks/midtrans`) — uses `request.text()` per Next.js convention
- Balance display as Server Component (no client JS)
- Spend action with error handling
- `.env.example` with every required variable and comments

### Smoke test

The example's README must include a "verify it works" section. In CI, we add a step that:

1. `cd examples/nextjs && pnpm install` (installs from npm, not workspace)
2. `pnpm build` (Next.js production build succeeds)

This catches broken imports if we rename/remove exports in a future version.

### Hono: docs-only, no separate example app

A "Hono Integration" code snippet in the docs guides section covers the API-only use case in ~30 lines. This avoids maintaining a second example while still showing backend-only usage with Xendit.

## What's explicitly NOT in v0.3.0

- No auto-generated API docs (TypeDoc) — hand-written is sufficient at this scale
- No custom domain — GitHub Pages default URL
- No Hono example app — covered by a code snippet in docs
- No i18n — English only
- No versioned docs — single version until v1.0
- No Algolia search — Pagefind (built into Starlight) is sufficient
- No Vercel deployment — GitHub Pages for simplicity

---

## Risk Mitigation

| Risk | Mitigation |
| --- | --- |
| Docs code examples drift from source | `?raw` imports from typechecked `.ts` files. CI breaks on drift. |
| Example app breaks on future releases | Smoke test in CI: `cd examples/nextjs && pnpm install && pnpm build`. |
| Scope creep into v0.4.0 features | Hard boundary: no new library features in v0.3.0. Docs and example only. |
| Starlight upgrade breaks docs | Pin `@astrojs/starlight` version. Upgrade intentionally, not automatically. |

## Success Criteria

1. A developer with no prior context can go from `pnpm add token-wallet` to a working Next.js app with balance, top-up, and spend in under 15 minutes by following the docs
2. `pnpm --filter @token-wallet/docs build` passes in CI and catches broken code examples
3. `cd examples/nextjs && pnpm install && pnpm build` succeeds in CI
4. All 13 docs pages are written and cross-linked
5. GitHub Pages deploys automatically on merge to main
